# Claude Code Setup

How to install Claude Code, point it at this repo, and operate the per-task
workflow. The roadmap is in `02_IMPLEMENTATION_STATUS.md`; the per-task
specs are in `03_IMPLEMENTATION_PLAN.md`. This file is operational.

---

## 1. Install

A paid Claude account is required (Pro minimum; Max recommended for heavy
sessions). The free Claude.ai plan does not include Claude Code.

**macOS / Linux (native installer, no Node.js needed):**
```bash
curl -fsSL https://claude.ai/install.sh | bash
```

**Homebrew (macOS):**
```bash
brew install --cask claude-code
```

**npm (any platform with Node 18+):**
```bash
npm install -g @anthropic-ai/claude-code
```

**Windows (PowerShell):**
```powershell
irm https://claude.ai/install.ps1 | iex
```

Verify:
```bash
claude --version
```

First run (`claude` from any directory) opens a browser auth flow.

Official docs (always more current than this file):
- Overview: https://docs.claude.com/en/docs/claude-code/overview
- Docs map: https://docs.anthropic.com/en/docs/claude-code/claude_code_docs_map.md

## 2. Open the Repo

```bash
git clone git@github.com:PedroCardoso-mdo/adflow.git
cd adflow
git checkout sa_gamma_rethetha
claude
```

`CLAUDE.md` auto-loads on session start. The numbered context files are NOT
auto-loaded — each task spec in `03_IMPLEMENTATION_PLAN.md` lists exactly
which files to load. This is intentional: token discipline.

## 3. Per-Task Workflow

For every task in the roadmap:

1. **Pick the next ❌ task** from the ordered list at the top of
   `02_IMPLEMENTATION_STATUS.md`. Do them in order.
2. **Open a fresh Claude Code session** (no leftover context from prior
   tasks).
3. **Send the task prompt:**
   ```
   @CLAUDE.md
   @02_IMPLEMENTATION_STATUS.md

   Do task <T-ID> from 03_IMPLEMENTATION_PLAN.md. Read only that one
   task block plus the files listed in its Context: line. Follow the
   per-task workflow in CLAUDE.md.
   ```
4. **Wait for one of three end-of-turn statuses:**
   - `STATUS: READY TO RUN` → run `python tests/smoke_transition.py`,
     confirm no NaN; `/clear`; next task.
   - `STATUS: TAPENADE NEEDED` → run the Tapenade regen commands
     (project-specific — keep them in a script), then
     `cd build && make -j`, then the smoke script; `/clear`; next task.
   - `STATUS: BLOCKED — <reason>` → read the reason, decide whether to
     refine the task spec or unblock manually.
5. **`/clear` between tasks.** Always. Stale context corrupts the next task.

---

## 4. Useful Slash Commands

| Command         | When to use                                          |
|-----------------|------------------------------------------------------|
| `/clear`        | Between tasks. Always.                               |
| `/compact`      | Mid-task if context fills past ~60%.                 |
| `/init`         | Only if `CLAUDE.md` is missing (it's not).           |
| `/permissions`  | Adjust auto-approval for bash, edits, etc.           |
| `/help`         | List all commands.                                   |

## 5. Recommended Project Settings

Save as `.claude/settings.json` (committed to git, shared with the team):

```json
{
  "permissions": {
    "allow": [
      "Bash(cd build && make*)",
      "Bash(grep *)",
      "Bash(rg *)",
      "Bash(find src*)",
      "Bash(git status)",
      "Bash(git diff*)",
      "Bash(git log*)",
      "Bash(git add*)",
      "Bash(git commit*)",
      "Read(src/**)",
      "Read(python/**)",
      "Read(*.md)",
      "Edit(src/turbulence/saGammaRetheta*)",
      "Edit(src/turbulence/turbBCRoutines.F90)",
      "Edit(src/modules/inputParam.F90)",
      "Edit(src/modules/paramTurb.F90)",
      "Edit(src/modules/block.F90)",
      "Edit(src/modules/blockPointers.F90)",
      "Edit(src/initFlow/initializeFlow.F90)",
      "Edit(src/utils/allocMem.F90)",
      "Edit(src/output/extraOutput.F90)",
      "Edit(src/modules/outputMod.F90)",
      "Edit(src/NKSolver/NKSolvers.F90)",
      "Edit(python/pyADflow.py)",
      "Edit(02_IMPLEMENTATION_STATUS.md)"
    ],
    "deny": [
      "Edit(src/adjoint/output*/**)",
      "Edit(src/modules/constants.F90)"
    ]
  }
}
```

The deny list is important. AD-generated files are regenerated by
Tapenade — Claude Code must NOT hand-edit them. The constants module is
mature; tasks that need new constants should add them in `paramTurb.F90`.

Personal settings (gitignored): `.claude/settings.local.json`.

## 6. Task Pacing Suggestion

The roadmap is 17 tasks. Most are 5-30 minutes of Claude Code time plus
the user's smoke run. Realistic per-day pace if you batch them:

- A1, A2, A3 — first session (foundation + visibility).
- B1, B2, B3, B4 — second session (bug fixes; B3 is the biggest, may be
  its own session).
- C1, C2, C3, C4 — third session.
- D1-D4 — fourth session (config-only, fast).
- E1, E2 — fifth session (AD regen, may need debugging).

Don't try to do all 17 in one day. Give yourself room to inspect output
between tasks.

---

## 7. Anti-Patterns

- **Loading all the numbered context files at session start.** They eat
  context and most tasks need only one or two sections. Let the task
  block tell you what to load.
- **Skipping `/clear` between tasks.** Past context contaminates the new
  task — Claude Code "remembers" decisions that were never properly
  committed and re-does work or skips prerequisites.
- **Letting Claude Code chain multiple tasks in one session.** Each task
  has a status row to flip and a commit. Chaining loses both.
- **Asking Claude Code to interpret physics results.** That is the user's
  job. Claude Code's job ends at compilation and (sometimes) successful
  smoke run.
- **Hand-editing AD-generated files.** Always regenerate via Tapenade.

---

## 8. When Things Go Wrong

- **Compile error Claude Code can't fix:** Output `STATUS: BLOCKED —
  <first error line>`, stop, hand it to the user.
- **Tapenade regen fails:** Not Claude Code's responsibility. User debugs
  the regen scripts.
- **Smoke run produces NaN after a task:** This is a regression. Open a
  new session, write a task using `06_TASK_TEMPLATE.md` describing the
  observed NaN and which task introduced it. Bisect via `git` if needed.
- **Two tasks seem entangled and you can't separate them:** Tell the user
  via `STATUS: BLOCKED — task X depends on task Y not yet done`. Do the
  prerequisite first.
